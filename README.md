# Fact-Checker Skill (事实核查技能) - 豆包/LLM 智能体部署指南

本技能（Skill）旨在指导大语言模型（LLM）对用户输入的言论进行**事实可靠性评估**。它能够识别用户话语中隐含的事实性陈述、引用数据或技术逻辑，并生成结构化的核查结果（`FactCheckResult`），帮助下游节点决定是接受、质疑、修正还是驳回该言论。

在**豆包（Doubao）平台 / 火山引擎智能体（Agent）** 或 RAG（检索增强生成）管道中，该技能可以作为**前置拦截器**或**内省思考层**使用。

---

## 🛠️ 如何在「豆包」或类似大模型平台中配置？

在豆包专业版（开发者平台）或配置自定义 Agent 时，你可以通过以下两种方式嵌入此技能：

### 方式一：直接写入"人设与回复逻辑（Prompt）"

如果你的智能体是单节点的，可以直接将以下中文核心逻辑复制到 Agent 的提示词（Prompt）中：

> 💡 **提示词植入模板：**
>
> 你内置了 `fact-checker` 技能。当用户提出明确的事实主张（如引用数据、阐述技术原理、提及政策法规、断言因果关系）时，你必须在内心（或输出前）进行事实核查：
>
> 1. **拆解观点**：将复杂的句子拆解为单一的事实断言。
> 2. **评估断言**：对比你的知识库，识别是否存在"以偏概全（部分属实）"、"时效过期"、"概念混淆"或"因果倒置"等错误。
> 3. **决定行动**：
>    - 如果完全正确 → 直接顺应回答 (`accept`)；
>    - 如果存在瑕疵/错误 → 自然地将修正信息融入对话中，不要生硬说教 (`correct`)；
>    - 如果涉及高风险（如中国移动支付最新政策、签证法规、医疗健康） → 必须结合联网搜索工具 (`search`)。

### 方式二：作为工作流（Workflow）的独立 Python/JSON 节点

在复杂的工作流中，你可以设置一个单独的 LLM 节点，专门负责输出 JSON。给该节点的 Prompt 如下：

```text
你是一个严谨的事实核查插件。请分析用户的输入，并严格按照以下 JSON 格式输出，不要包含任何推理正文或 Markdown 代码块包裹：

{
  "claim": "抽取出的用户核心断言",
  "verdict": "true | mostly_true | uncertain | mostly_false | false | unverifiable",
  "confidence": "high | medium | low",
  "reasoning": "2-4句中文解释判定理由",
  "error_type": null 或 "factual_error | conflation | causal_error | anachronism | magnitude_error | partial_truth",
  "recommended_action": "accept | clarify | correct | escalate | search"
}
```

## 📖 核心使用规范（中文版）

### 1. 什么时候触发这个技能？

- **触发场景**：用户说了具体的事实（如："外籍人士在中国不能用微信支付"、"中国早就全面禁止 VPN 了"）。
- **不触发场景**：纯主观观点（"我觉得微信不好用"）、假设性提问（"如果人类能飞会怎么样？"）、用户本身就高度怀疑的话（"我不确定是不是这样……"）。

### 2. 核心判定标准 (`verdict`)

| verdict | 含义 |
|---|---|
| `true` | 完全属实：完全符合当前已知事实。 |
| `mostly_true` | 基本属实：核心是对的，但有细节错误或带有误导性。（这是最常见也是最危险的"部分属实"情况） |
| `mostly_false` | 基本捏造：存在严重事实错误，但并非全盘皆错。 |
| `false` | 完全错误：纯属谣言或常识性错误。 |
| `unverifiable` | 无法验证：涉及他人隐私、未来的预测或无法证伪的言论。 |

### 3. 特殊高危区域：针对中国本土场景的特殊处理

在中国数字化生态（SinoEZ 或跨境旅游/业务）中，以下五大领域属于**极易由于训练集过期而产生幻觉的"高危区"**。一旦触发，推荐动作（`recommended_action`）必须强制设为 `search`（联网搜索）：

- **移动支付与绑定**：海外信用卡（Visa/Mastercard）绑定微信/支付宝的最新额度、手续费及身份认证政策（更新极快）。
- **免签与签证政策**：各国对华、或中国对各国的最新免签口岸、停留天数、过境免签（144小时）政策。
- **网络与合规**：哪些海外 App 在国内可以直连，VPN 的法律边界（注意：作为 Agent **严禁提供任何翻墙教程或法律建议**，应提示风险并建议遵守当地法律）。
- **企业与高管信息**：国内大厂的组织架构变动、高管更迭。
- **出行与生活常识**：高铁购票新规（如外籍护照线上核验）、无人驾驶出租车（如萝卜快跑）的落地城市和运营范围。

## 🎯 话术示例：如何在豆包中做到"自然修正"

在日常对话模式中（Mode 1: Silent），模型**不需要**把冷冰冰的 JSON 甩给用户，而是应该把核查结果自然地变成高情商的回复。

### ❌ 反面教材（说教式回复）：

> "错误。你的断言'外国人不能用微信支付'属于 `partial_truth` 错误。根据2023年腾讯新规，外国人是可以绑定的。你的信息落后了。"

*Grant（情商低，像个刻板的审查员）*

### ✅ 正面教材（自然融入式回复）：

> "以前确实是这样，不过**现在外国人去中国旅游方便多了**！从2023年开始，微信和支付宝已经支持直接绑定境外的 Visa 或 Mastercard 信用卡了。
>
> 需要注意的是（风险提示）：虽然可以绑定，但在一些小城镇或者街边摊，外卡支付偶尔会遇到风控或刷不过去的情况。建议你还是准备一点少量的人民币现金，或者下载一个'数字人民币' App 作为备用。你想了解具体怎么绑定吗？"

*Empathy & Candor（既肯定了用户的历史认知，又纠正了现状，同时给出了实用的出行建议）*

## 📈 进阶：多轮对话信用分 (`ClaimLog`)

在豆包的工作流或 Memory（记忆）模块中，建议维护一个简单的用户画像：

- 如果用户在连续 **3 轮对话**中提供的信息都是 `false` 或 `mostly_false`（例如：一直在用过期的陈旧信息质问 Agent）。
- 系统应自动调高核查级别，将 `overall_reliability_signal` 设为 `low`（低信用度）。在接下来的对话中，**只要用户抛出论据，模型必须优先进行联网检索，不再盲信用户提供的任何上下文**。
